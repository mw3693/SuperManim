# PART ONE — ANSWER TO QUESTION ONE
## "Can I divide the domain services like `timeline_service` and `hash_service`,
 or should I keep each one as a single file?"

**Short answer: Keep each domain service as ONE single file. Do not divide them.**

Here is why, explained in simple terms.

---

### What is a Domain Service?

A domain service is a file that lives inside your `core/services/` folder.
It contains **pure logic only** — no database calls, no file reading, no FFmpeg, nothing external.

Think of it like a math textbook.
A math textbook only contains math rules. It does not contain a printer, a phone,
or a database connection. It is just clean knowledge.

Your domain services are the same. They only contain the rules of your project.

---

### The `timeline_service.py` — What Does It Do?

The `timeline_service.py` is responsible for one single topic: **time and sequence**.

Everything that involves time logic belongs inside this one file. Examples:

**Function 1: `calculate_scene_duration(start_time, end_time)`**
- You give it a start time and an end time.
- It gives you back the duration in seconds.
- This is a pure calculation. No database needed.

**Function 2: `validate_timeline_has_no_gaps(scenes)`**
- You give it a list of scenes with their start and end times.
- It checks: is there any empty time gap between scenes?
- Example: Scene 1 ends at 12.5 seconds. Scene 2 starts at 14.0 seconds.
  That is a gap of 1.5 seconds. This function finds that and reports it.

**Function 3: `validate_timeline_has_no_overlaps(scenes)`**
- You give it a list of scenes.
- It checks: do any two scenes overlap in time?
- Example: Scene 1 covers 0 to 15 seconds. Scene 2 covers 13 to 25 seconds.
  They overlap at 13–15 seconds. This function finds that.

**Function 4: `build_timeline_from_scenes(scenes)`**
- You give it a list of scenes.
- It builds a full ordered timeline object that shows the complete sequence.

**Function 5: `reorder_scenes_by_start_time(scenes)`**
- You give it scenes in any order.
- It sorts them by start time and returns them in the correct order.

**Function 6: `split_timeline_at_position(timeline, position_seconds)`**
- You give it a timeline and a time position (e.g., 30.0 seconds).
- It splits the timeline into two halves: before 30 seconds and after 30 seconds.

All of these functions are about **time and sequence**. They all belong in ONE file
called `timeline_service.py`. There is no reason to split them into smaller files,
because they all deal with the same topic. Splitting them would make your project
harder to navigate, not easier.

---

### The `hash_service.py` — What Does It Do?

The `hash_service.py` is responsible for one single topic: **fingerprinting**.

A fingerprint (called a hash) is a short code that uniquely identifies a file's content.
If you change even one character in a file, the hash completely changes.
SuperManim uses hashes to know whether a scene has changed since last time.

**Function 1: `compute_file_hash(file_path) -> str`**
- You give it a path to a file on disk.
- It reads the file content and runs the SHA-256 algorithm on it.
- It returns a long string like `"a3f8b2c..."`.
- Same file content = always same hash. Changed file = completely different hash.

**Function 2: `compute_string_hash(text_content) -> str`**
- Same idea, but instead of a file path, you give it a text string directly.
- Useful for hashing small pieces of data without creating a file.

**Function 3: `hashes_are_equal(hash_one, hash_two) -> bool`**
- You give it two hash strings.
- It returns True if they are equal, False if not.
- Simple comparison, but having it as a named function makes the code readable.

**Function 4: `scene_has_changed(old_hash, new_hash) -> bool`**
- A named wrapper around the comparison above.
- Returns True if the scene's fingerprint changed (meaning the scene needs re-rendering).
- Makes the calling code read like English: `if scene_has_changed(old, new): render it`

All of these functions are about **computing and comparing fingerprints**.
They all belong in ONE file called `hash_service.py`.

---

### The Rule To Remember

> If all the functions in a service file are about the same topic, keep them in one file.
> Only split a file when it has grown so large (200+ lines) that you cannot read it
> comfortably, AND when two clearly separate topics have accidentally ended up
> in the same file.

Your `timeline_service.py` and `hash_service.py` are both focused on exactly one
topic each. Keep them as single files. Do not divide them.

# PART TWO — ANSWER TO QUESTION TWO
## "Are there any Application Services related to timeline and hash, and if yes, what are they?"

**Short answer: Yes. There are TWO application services that use the domain services.**

But first, let us understand the difference between a Domain Service and an Application Service,
because this is the most important thing to understand.

---

### Domain Service vs Application Service — The Simple Difference

```
DOMAIN SERVICE (lives in core/services/)
─────────────────────────────────────────────────────────────────────
This is a PURE MATH FUNCTION FILE.
It takes data in. It does logic. It returns a result.
It NEVER talks to a database.
It NEVER reads a file from disk.
It NEVER calls an external tool.
It just does calculation and returns answers.

Example: timeline_service.compute_scene_duration(start=0, end=12.5)
Returns: 12.5
It did math. Nothing else.


APPLICATION SERVICE (lives in application/use_case/)
─────────────────────────────────────────────────────────────────────
This is a WORKFLOW COORDINATOR FILE.
It USES the domain service to get answers.
It ALSO talks to repositories to load and save data.
It ALSO calls other ports to do real work.
It is the "manager" that connects all the pieces together.

Example: render_service.render_scene(scene_id)
Step 1: Load scene from database (uses repository)
Step 2: Compute scene's current hash (uses hash domain service)
Step 3: Load old hash from cache (uses cache repository)
Step 4: Compare hashes (uses hash domain service again)
Step 5: If different → call Manim (uses render runner port)
Step 6: Save new hash to cache (uses cache repository)
Step 7: Report progress (uses progress reporter port)
```

The domain service only does one small step (step 2 and step 4 above).
The application service does the entire workflow from step 1 to step 7.

---

### Application Service That Uses `hash_service`: `render_service.py`

This already exists in your project. Here is what it does and why it needs `hash_service`:

`render_service.py` (located in `application/use_case/render_service.py`) is the
service that decides which scenes to render and which to skip.

**Its full workflow looks like this:**

```
User runs: supermanim render --project MyAnimation

RenderService does:
1. Load all scenes from project_data.db
2. For each scene:
   a. Read the scene's Python file from disk
   b. Compute a hash of that file using hash_service.compute_file_hash()
   c. Load the stored hash from the cache_records table in the database
   d. Compare the two hashes using hash_service.hashes_are_equal()
   e. If EQUAL   → scene has not changed → SKIP → use the cached video
   f. If DIFFERENT → scene has changed   → CALL MANIM → render it
   g. Save the new hash to cache_records after rendering
3. Assemble all video clips in order using the timeline
4. Merge with audio if present
5. Export the final video
```

So `render_service.py` calls `hash_service.py` at steps 2b and 2d.
Without the hash service, the render service would have to contain the
hashing logic itself, which would break the separation of concerns.

---

### Application Service That Uses `timeline_service`: `sync_service.py`

This already exists in your project. Here is what it does and why it needs `timeline_service`:

`sync_service.py` (located in `application/use_case/sync_service.py`) is the
service that keeps all the audio clips and scenes perfectly time-aligned.

**Its full workflow looks like this:**

```
User adds a new audio clip or changes a scene duration.

SyncService does:
1. Load all scenes and audio clips from the database
2. Build the current timeline using timeline_service.build_timeline_from_scenes()
3. Check for gaps using timeline_service.validate_timeline_has_no_gaps()
4. Check for overlaps using timeline_service.validate_timeline_has_no_overlaps()
5. If problems found → report them to the user
6. If no problems → confirm everything is aligned
7. Update the database with the validated timeline
```

So `sync_service.py` calls `timeline_service.py` at steps 2, 3, and 4.

---

### Also: `scene_service.py` Uses Both

`scene_service.py` (in `application/use_case/scene_service.py`) also uses both.

When you add a new scene or change a scene's time range:
- It uses `timeline_service` to validate that the new scene fits without overlap.
- It uses `hash_service` to compute and store the initial hash of the scene file.

---

### Summary of the Connections

```
hash_service.py (domain)
    └── used by → render_service.py (application)
    └── used by → scene_service.py (application)

timeline_service.py (domain)
    └── used by → sync_service.py (application)
    └── used by → scene_service.py (application)
    └── used by → export_service.py (application) [when assembling final video order]
```

You do NOT need to create new application services just for hash or timeline.
The ones you already have are the correct ones. You just need to make sure
`render_service`, `sync_service`, `scene_service`, and `export_service`
import and use these domain services when they need them.

# PART THREE — ANSWER TO QUESTION THREE
## "What helper functions should I put in the `utils/` folder? Explain each one in simple terms."

The `utils/` folder is for **general tools that have no business logic**.
Think of it as your toolbox. A hammer in a toolbox does not care whether you are
building a house or fixing a fence. It just hammers nails.

Your utils functions are the same. They do not care about scenes, projects, or audio.
They just do small, reusable jobs that many parts of the project need.

Here is every useful function you should put in `utils/utilities.py`, each explained simply:

---

### Group 1 — Path and File Name Helpers

**`sanitize_filename(name: str) -> str`**

Problem: A user types a project name like `"My Project / 2024 : Test!"`.
That name has slashes, colons, and exclamation marks. Those characters are
illegal in file and folder names on Windows and some Linux systems.

This function takes any string and removes or replaces illegal characters.
It turns `"My Project / 2024 : Test!"` into `"My_Project_2024_Test"`.

You use this every time you create a new project folder from a user-typed name.

---

**`ensure_directory_exists(path: str) -> None`**

Problem: Before you can save a file to a folder, that folder must exist.
If it does not exist, your save will crash.

This function checks if a folder exists. If it does not exist, it creates it.
If it already exists, it does nothing.

You use this before creating any project folder, any `output/` folder, any `cache/` folder.

---

**`get_file_extension(file_path: str) -> str`**

Problem: You have a file path like `/home/user/audio/narration.mp3`.
You need to know whether it is an `.mp3`, `.wav`, or `.aac` file.

This function takes any file path and returns just the extension.
Input: `/home/user/audio/narration.mp3`
Output: `"mp3"`

---

**`get_filename_without_extension(file_path: str) -> str`**

Similar to above. Returns just the base name without extension.
Input: `/home/user/audio/narration.mp3`
Output: `"narration"`

---

**`build_output_path(project_path: str, filename: str) -> str`**

Takes a project path and a filename and correctly builds the full output path.
Input: `"/projects/MyAnimation"`, `"scene_01.mp4"`
Output: `"/projects/MyAnimation/output/scene_01.mp4"`

Saves you from writing `os.path.join()` and manually adding `"output"` everywhere.

---

### Group 2 — Time and Duration Helpers

**`seconds_to_hhmmss(seconds: float) -> str`**

Problem: You have a duration of `3723.5` seconds. That is hard to read.

This function converts seconds to a human-readable time string.
Input: `3723.5`
Output: `"01:02:03.500"` (1 hour, 2 minutes, 3.5 seconds)

You use this when displaying render times and audio durations to the user.

---

**`hhmmss_to_seconds(time_string: str) -> float`**

The reverse of the function above.
Input: `"01:02:03.500"`
Output: `3723.5`

You use this when a user types a time in the CLI and you need to store it as a number.

---

**`round_duration(seconds: float, decimal_places: int = 3) -> float`**

Problem: Floating point math in Python can produce numbers like `12.500000000001`.
This function rounds them cleanly to 3 decimal places.
Input: `12.500000000001`
Output: `12.5`

Very important for audio sync. Even tiny floating point errors in timestamps can
cause small audio/video misalignments.

---

**`durations_are_close_enough(duration_a: float, duration_b: float, tolerance: float = 0.01) -> bool`**

Problem: A scene is supposed to be exactly `12.5` seconds long, but after rendering
it comes out as `12.497` seconds. Is that acceptable, or is it a problem?

This function checks whether two durations are "close enough" within a tolerance.
Input: `12.497`, `12.5`, tolerance `0.01`
Output: `True` (they are within 0.01 seconds of each other, so it's fine)

You use this to check if rendered scene length matches expected length.

---

### Group 3 — String Formatting Helpers

**`format_scene_display_name(scene_order: int, scene_name: str) -> str`**

Takes a scene number and name and formats it nicely for display.
Input: `1`, `"Introduction"`
Output: `"Scene 01 — Introduction"`

Used in CLI output so the user sees consistent, readable scene names.

---

**`truncate_string(text: str, max_length: int, suffix: str = "...") -> str`**

Problem: A file path might be very long. When printing it in the terminal,
it goes off the screen and looks ugly.

This function cuts a string at a maximum length and adds `"..."` at the end.
Input: `"/very/long/path/to/project/MyAnimation/scenes/scene_01.py"`, max `40`
Output: `"/very/long/path/to/project/MyAnimat..."`

---

**`pluralize(count: int, singular: str, plural: str) -> str`**

Problem: You want to print `"1 scene"` but also `"3 scenes"` correctly.

This function picks the right word automatically.
Input: `1`, `"scene"`, `"scenes"` → Output: `"1 scene"`
Input: `3`, `"scene"`, `"scenes"` → Output: `"3 scenes"`

---

### Group 4 — Validation Helpers

**`is_valid_project_name(name: str) -> bool`**

Checks if a project name is valid (not empty, not too long, no illegal characters).
Returns True or False.

Used by `project_service.py` before creating a project.

---

**`is_supported_audio_format(file_path: str) -> bool`**

Checks if the file extension is one of the supported audio formats (mp3, wav, aac, flac).
Returns True or False.

Used by `audio_service.py` before accepting an audio file from the user.

---

**`is_supported_video_format(file_path: str) -> bool`**

Same idea, but for video formats (mp4, avi, mov, mkv).

---

**`is_file_readable(file_path: str) -> bool`**

Checks if a file exists AND can be read (not corrupted, not permission-blocked).
Returns True or False.

Used before passing any file to FFmpeg or Manim. If you pass an unreadable file
to FFmpeg it crashes with a confusing error. This check gives a clear message instead.

---

### Group 5 — List and Data Helpers

**`chunk_list(items: list, chunk_size: int) -> list`**

Problem: You have 20 scenes and you want to process them in batches of 5.

This function splits any list into smaller groups.
Input: `[1,2,3,4,5,6,7]`, chunk_size `3`
Output: `[[1,2,3], [4,5,6], [7]]`

Useful if you ever want to render multiple scenes in parallel batches.

---

**`find_duplicates(items: list) -> list`**

Takes a list and returns any values that appear more than once.
Input: `["scene_01", "scene_02", "scene_01", "scene_03"]`
Output: `["scene_01"]`

Useful for catching duplicate scene names before adding them to the project.

---

**`flatten_list(nested_list: list) -> list`**

Takes a list of lists and returns a single flat list.
Input: `[[1,2,3], [4,5], [6]]`
Output: `[1, 2, 3, 4, 5, 6]`

---

### Group 6 — Date and Time Stamp Helpers

**`get_current_timestamp() -> str`**

Returns the current date and time as a string in a standard format.
Output: `"2024-11-15 14:32:07"`

Used when recording when a project was created, when a scene was last rendered, etc.

---

**`timestamp_to_display_string(timestamp: str) -> str`**

Takes a stored timestamp and converts it to a friendly readable format.
Input: `"2024-11-15 14:32:07"`
Output: `"November 15, 2024 at 2:32 PM"`

Used when showing project history or render history to the user.

---

# PART FOUR — ANSWER TO QUESTION FOUR
## "Should I make one `database_connection_service` as an application service to manage both
`session.db` and `project_data.db`? Or should I make two separate files?"

**Short answer: Make ONE connection manager, but keep it in the `database/` folder as a
plain helper class — NOT as an application service.**

Here is the full explanation.

---

### First: Why You Should NOT Make Two Separate Files

If you make two completely separate files like `session_connection.py` and
`project_connection.py`, you have this problem: every time you need to fix
a bug in the connection logic (for example, enabling WAL mode in SQLite for
better performance), you must fix it in two places. Then later you might fix
it in one file and forget the other. This is a well-known programming trap.

The rule is: **do not repeat the same logic in two places.**

---

### Second: Why You Should NOT Make It an Application Service

An **Application Service** (in `application/use_case/`) is for **workflows**.
It coordinates steps. It uses ports. It calls domain services.

A **database connection** is not a workflow. It is a simple technical tool.
It is infrastructure. It has no business logic. It does not call domain services.
It does not validate data. It just opens and returns a SQLite connection.

Putting it in `application/use_case/` would be like putting a screwdriver
inside a recipe book. It does not belong there.

---

### The Correct Solution: One Manager in the `database/` Folder

You already have a `database/` folder with `connection.py` inside.
This is exactly the right place. Here is how to design it well.

**File: `database/connection.py`**

This file should contain ONE class called `DatabaseConnectionManager`.

Here is what it does, step by step:

**Step 1 — It knows about both databases:**
```
session_db_path    = ~/.supermanim/session.db
project_db_path    = <current_project_folder>/project_data.db
```

**Step 2 — It gives you a connection when you ask for it:**
```python
manager = DatabaseConnectionManager()

# To get a session connection:
conn = manager.get_session_connection()

# To get a project connection:
conn = manager.get_project_connection("/projects/MyAnimation")
```

**Step 3 — It applies the same settings to every connection it creates:**
- Enables WAL (Write-Ahead Logging) mode — makes SQLite faster and safer
- Sets a timeout — so if two things try to write at the same time, it waits instead of crashing
- Enables foreign key enforcement — so your data stays consistent
- Returns rows as dictionaries instead of plain tuples — so you can use `row["scene_name"]`
  instead of `row[0]`, which is much easier to read

**Step 4 — It handles errors in one place:**
If the database file cannot be opened (file is corrupted, path is wrong),
it raises a clear error message from one place. You do not repeat error
handling in every repository file.

---

### How the Repositories Use This Manager

Each SQLite repository adapter imports and uses the manager:

```
sqlite_scene_repository.py
    → asks manager: "give me a connection to project_data.db"
    → manager returns a connection
    → repository uses it to run SQL
    → repository closes the connection when done

sqlite_session_repository.py
    → asks manager: "give me a connection to session.db"
    → manager returns a connection
    → repository uses it
    → repository closes when done
```

The repositories do NOT contain any connection settings.
They just ask the manager for a connection and use it.
All connection settings live in ONE place only: `connection.py`.

---

### Final Design of `database/connection.py`

```
DatabaseConnectionManager
│
├── get_session_connection() -> sqlite3.Connection
│     Opens (or creates) session.db
│     Applies standard settings (WAL, timeout, row_factory, foreign_keys)
│     Returns the ready connection
│
├── get_project_connection(project_path: str) -> sqlite3.Connection
│     Opens (or creates) project_data.db inside the given project folder
│     Applies the same standard settings
│     Returns the ready connection
│
└── _apply_standard_settings(connection) -> None
      Private helper called by both methods above
      Sets WAL mode, timeout, row_factory, foreign_keys
      Lives in one place only
      Changes here automatically apply to both databases
```

---

### Why This Is Better Than the Alternatives

```
OPTION A: Two separate files (session_connection.py + project_connection.py)
─────────────────────────────────────────────────────────────────────────────
PRO:  Simple to understand.
CON:  Duplicate logic in two places.
CON:  When you fix a bug, you must remember to fix it twice.
CON:  As the project grows, the two files drift apart.
VERDICT: NOT recommended.


OPTION B: DatabaseConnectionService as an Application Service
─────────────────────────────────────────────────────────────────────────────
PRO:  Everything in one place.
CON:  Wrong layer. Application services are for workflows, not infrastructure.
CON:  Confuses the architecture. Future developers will be confused by it.
CON:  Violates Hexagonal Architecture principles.
VERDICT: NOT recommended.


OPTION C: DatabaseConnectionManager in database/connection.py (RECOMMENDED)
─────────────────────────────────────────────────────────────────────────────
PRO:  One place for all connection logic.
PRO:  Correct layer — infrastructure belongs in its own folder.
PRO:  Easy to extend — add a third database later by adding one method.
PRO:  All repositories get the same consistent settings.
PRO:  Clean architecture — each layer has the right things in it.
VERDICT: BEST choice. This is what you already started with in database/connection.py.
         Just design it as shown above.
```

# FINAL SUMMARY — THE FOUR ANSWERS IN ONE PARAGRAPH EACH

**Question 1:** Keep `timeline_service.py` and `hash_service.py` each as one single file.
Each file has one clearly focused topic (time logic / fingerprinting logic).
Only split a file when two completely unrelated topics accidentally ended up in the same file.

**Question 2:** The application services that use the domain services are:
`render_service.py` uses `hash_service`, and `sync_service.py` + `export_service.py`
use `timeline_service`. `scene_service.py` uses both. You do not need to create
any new application services. Just make sure the existing ones import and call
the domain services when they need them.

**Question 3:** The `utils/utilities.py` file should contain 22 helper functions organized
into 6 groups: Path/File helpers, Time/Duration helpers, String formatting helpers,
Validation helpers, List/Data helpers, and Timestamp helpers. None of these functions
contain business logic. They are all reusable tools that many parts of the project need.

**Question 4:** Use ONE class called `DatabaseConnectionManager` inside `database/connection.py`.
It handles both `session.db` and `project_data.db`. It applies the same settings to both.
Do not make it an Application Service — it is infrastructure, not a workflow.
All repository adapters ask this manager for a connection instead of each one
building its own connection from scratch.
